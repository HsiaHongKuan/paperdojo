## Code-Level Findings (from `generate_report.py` analysis)

**4 issues found in the script, 0 critical:**

1. **`load_analysis` has no error handling (medium).** If the subagent wraps JSON in markdown code fences (common LLM failure mode), `json.loads` crashes. The prompt says "Return ONLY the JSON object" but this isn't enforced programmatically. File: `scripts/generate_report.py`, line ~58.

2. **`render_glance` malformed text in history-only case (minor).** When feeds are empty but history exists, `parts[:3]` is empty but `parts[3]` (coaching stats) exists. The join produces `. Of your 2 coaching sessions...` with a leading period-space. File: `generate_report.py`, lines ~258-283.

3. **Word cloud double-counts coached papers (minor).** `compute_word_cloud` collects titles from both explored feed papers and history sessions independently. Papers appearing in both get inflated word counts. Negligible with few sessions, grows with scale. File: `generate_report.py`, lines ~124-137.

4. **Stale comment on heatmap border color (trivial).** Line ~219 says `# gold` but the color is `#3b82f6` (blue). Comment wasn't updated when color changed.

## Structural Observations

1. **Single-file HTML is a strong design choice.** No build step, no dependencies, no server. Opens in any browser, can be emailed or embedded.

2. **Data pipeline is well-defined but tightly coupled.** The report reads from feeds/ and history/ which are written by /feed and /coach. The JSON schemas match but aren't formally documented. If either skill changes its output format, the report breaks silently.

3. **No incremental/diff reporting.** Each invocation regenerates from scratch. Fine for dozens of sessions, won't scale to hundreds without date-range filtering.

4. **Analysis subagent prompt is well-specified.** It asks for specific JSON keys (at_a_glance, coaching_patterns, best_moments, stuck_points, topics_to_revisit, research_directions) with markdown format. The "Return ONLY the JSON object" instruction prevents conversational padding.

5. **Step 2 is skipped when no history exists.** The spec correctly handles the feeds-only case (stats and heatmap without narrative sections). But the transition from "no history" to "some history" isn't explicitly noted — the first time coaching insights appear could surprise the user.

## Suggestions

1. **High impact: Add date-range parameter.** Allow `/report --last 7d` or `/report --month 2026-03` to scope the dashboard.

2. **High impact: Explicit shared schema.** Document the session JSON fields in a shared location (or as comments in both /coach and /report SKILL.md files) to prevent silent format drift.

3. **Medium impact: Insight thresholds.** State minimum data requirements for coaching insights (e.g., "Patterns require at least 5 sessions"). Below threshold, show "not enough data" instead of generic encouragement.

4. **Medium impact: Add `open` command to spec.** The spec includes `open .paperdojo/report.html` for macOS — this is a nice UX touch that should be preserved.

5. **Low impact: Accessibility.** The generated HTML should include alt text on heatmap cells and ARIA labels on stats cards for screen reader support.
