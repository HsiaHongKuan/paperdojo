## Structural Observations

### 1. No back navigation (CRITICAL)
The skill offers no [b] option to revisit a previously shown paper. This is a significant UX gap for a sequential browsing interface. Marcus triggered this exact scenario -- skipping a paper and immediately wanting to go back. The current workaround is to quit and re-run the feed, but skipped papers are recorded and would be excluded from future feeds (they are written to the feed JSON with "skipped" action and collected as "previously seen arxiv IDs"). This means a hasty [n] is effectively irreversible for the current day.

### 2. No "save for later" action (CRITICAL)
The only actions are skip, details, coach, or quit. There is no lightweight "bookmark" or "save" action for papers the user finds interesting but doesn't want to coach on right now. The [q] action records the current paper as "seen" but this is implicit and doesn't distinguish between "I saw this and it's meh" vs "I saw this and want to come back." This forces users into a false binary: either commit to a full coaching session or lose the paper to the "skipped"/"seen" pile.

### 3. [q] records current paper without user decision
When the user hits [q], the currently displayed paper is recorded as "seen" automatically. The user gets no say in whether this paper should be "skipped" or "interested." Marcus quit while viewing paper 3 (which he found highly relevant), and it was recorded as merely "seen" -- the same status it would get if he found it boring. The skill should either prompt for a final action on the current paper or use a distinct status like "quit_on" to preserve the signal.

### 4. MCP install prompt interrupts browsing flow
The MCP availability check happens mid-browse when the user requests details. This is a jarring context switch from paper evaluation to system administration. The persona found this frustrating. Better to check at session start (Step 2) and set a flag, or always use the fallback without prompting.

### 5. Feed summary lacks granularity
The summary reports "N papers browsed, M skipped, K coached" but does not mention papers where details were viewed, or how many remain unviewed. Useful additions: "L details viewed" and "P papers remaining in today's feed."

### 6. No paper dates in presentation format
The display format includes `{date}` in the template but the fetch step does not explicitly extract publication date. The ranking subagent output format also omits date. Marcus specifically noted wanting to confirm papers are recent.

### 7. Feed file append strategy may cause corruption
The skill reads the JSON file, appends to the array, and writes back on every [n]/[s]/[q] action. If the session is interrupted mid-write, the file could be corrupted. A safer approach would be to buffer all records in memory and write once at session end, or use append-only JSONL format.

### 8. No explicit handling of empty fetch results
The skill says "If the subagent returns no papers, ask the user what they'd like to do" but provides no structured guidance. This could lead to an awkward open-ended conversation when the user expects a deterministic tool.

### 9. Previously-seen deduplication reads ALL feed files
Step 2 says "Read all files in `.paperdojo/feeds/`" to collect seen IDs. Over months of daily use, this directory could accumulate hundreds of files. The skill should either use a separate seen-IDs index file or only read recent files (e.g., last 30 days).

## Suggestions

Ordered by impact:

1. **Add [b] back navigation** -- Allow the user to type [b] to revisit the previous paper. Maintain a simple stack of shown papers. When going back, re-display the paper with its original prompt options (minus [d] if details were already viewed). This is the most requested feature from the persona and addresses the irreversibility of [n]. *(HIGH IMPACT, LOW EFFORT)*

2. **Add [f] favorite / save-for-later action** -- Add a lightweight bookmark action that records `"action": "saved"` in the feed JSON. This gives users a middle ground between skipping and coaching. Saved papers could be surfaced in future `/feed` sessions or a dedicated `/saved` view. *(HIGH IMPACT, LOW EFFORT)*

3. **Check MCP availability at session start** -- Move the arxiv-latex-mcp availability check to Step 2 (after fetch, before browse). Set a `mcp_available` flag. If not available, either silently use the fallback for all [d] requests, or ask once at the start instead of mid-browse. *(HIGH IMPACT, LOW EFFORT)*

4. **Prompt for final action on [q]** -- When the user quits, ask what to do with the currently displayed paper: "Before you go -- mark this paper as [s] save / [k] skip / [done] just quit?" This preserves user intent instead of silently recording "seen." *(MEDIUM IMPACT, LOW EFFORT)*

5. **Include paper dates in display and fetch** -- Add `published_date` or `submitted_date` to the fetch subagent's output format and display it in the paper card. For a daily feed, date is essential context. *(MEDIUM IMPACT, LOW EFFORT)*

6. **Add "show remaining titles" option** -- Add [l] list to show titles of remaining unviewed papers. This lets users make an informed decision about whether to continue browsing or quit. *(MEDIUM IMPACT, LOW EFFORT)*

7. **Buffer feed records and write once** -- Instead of read-append-write on every action, accumulate records in memory and write the complete feed JSON once at session end (on [q] or [s]). This prevents file corruption from interrupted sessions. *(MEDIUM IMPACT, LOW EFFORT)*

8. **Cap previously-seen deduplication window** -- Only read feed files from the last 30 days for deduplication, or maintain a separate `seen_ids.json` index that gets appended to. This prevents performance degradation over time. *(LOW IMPACT, MEDIUM EFFORT)*

9. **Add feed summary granularity** -- Report details-viewed count and remaining-papers count in the quit summary. E.g., "Feed session complete. 3 of 5 papers browsed: 2 skipped, 1 viewed in detail. 2 papers remaining." *(LOW IMPACT, LOW EFFORT)*
