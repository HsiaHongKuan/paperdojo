## Structural Observations

### Intent vs Experience by Phase

| Phase | Skill Intent | User Experience | Gap |
|-------|-------------|-----------------|-----|
| Step 1 | Verify config, fallback to inline setup | Seamless — config existed, no friction | None |
| Step 2 | Fetch and rank by relevance | Ranking felt appropriate; most relevant first | Fetch is opaque (no loading indicator) |
| Step 3 | One-at-a-time browse with action choices | Linear flow works for scanning but frustrates comparison | No back navigation or bookmarking |

### Design Observations

1. **No back navigation.** The browse loop is strictly forward. Once [n] is pressed, the paper is gone. The persona explicitly wanted to revisit. The [s] decision often requires comparing across papers, which the current flow doesn't support.

2. **[s] and [q] are mutually exclusive terminals.** Correct behavior, but means comprehensive testing requires two separate runs.

3. **MCP detection is unspecified.** Step 2 says "Fast path (arxiv-mcp-server installed)" without specifying how to detect availability. Should say: "attempt to call the MCP tool; if it fails, use the fallback."

4. **[d] triggers MCP install mid-browse.** If arxiv-latex-mcp isn't installed, the first [d] press interrupts browsing with an install prompt. This is jarring — could check at session start instead.

5. **"seen" vs "skipped" semantics.** On [q], the displayed paper is recorded as `"action": "seen"`. The distinction from "skipped" (actively chose next) is subtle and may not be meaningful for reporting.

6. **Post-details prompt correctly drops [d].** After viewing details, re-prompt shows only `[n] [s] [q]`. Prevents redundant views.

## Suggestions

1. **High impact: Add a bookmark action** (e.g., `[b] Bookmark`). Record `"action": "bookmarked"` in feed file. At session end, show bookmarked papers and let user pick one for coaching. Addresses the comparison problem without breaking one-at-a-time flow.

2. **High impact: Check MCP at session start**, not on first [d] press. Prompt install once upfront or note that details will use abstract analysis. Avoids mid-browse interruption.

3. **Medium impact: Specify MCP detection** in SKILL.md. Add: "To check MCP availability, attempt to call the tool. If it fails with 'tool not found', it is not available."

4. **Medium impact: Consider summary-first mode** as alternative to sequential browse. Show ranked title list with one-line descriptions, let user pick which to expand. Serves expert users who scan quickly.

5. **Low impact: Add [q] test as separate run** in any test suite. Since [s] and [q] are mutually exclusive, comprehensive testing requires two runs with different persona tendencies.
