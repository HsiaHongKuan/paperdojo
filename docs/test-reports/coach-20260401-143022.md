## Structural Observations

### 1. CRITICAL: No pause/resume capability
The coaching session has no mechanism for the user to pause, save progress, and return later. For complex papers that require offline thinking (working through algebra, drawing diagrams, consulting references), this is a significant gap. The session is synchronous and all-or-nothing: you either reach the reveal in one sitting or lose your progress.

**Recommendation:** Add a `[p]` pause option that saves the session state with `outcome: "paused"` and resumes when the user runs `/coach 2603.19847` again and the history file exists with a paused state.

### 2. CRITICAL: No criteria for "solved" outcome
The skill defines `outcome: "solved"` for when the user arrives at the solution, but provides no criteria for when to declare this. Questions that need answers:
- How close does the user's approach need to be? Exact method name? Same key insight? Same asymptotic complexity?
- Who decides -- the coach (risk of false positives/negatives) or the user ("I think I've got it")?
- The trigger phrase "I think I've got it" is listed as a reveal trigger, not a "solved" declaration. There's no separate mechanism for the user to claim they've solved it and have the coach validate.

**Recommendation:** When the user says "I think I've got it," ask them to state their approach explicitly. Then assess: if they've identified the key insight(s), mark as "solved" and reveal for confirmation. If not, offer another hint or reveal. This makes the solved/revealed distinction meaningful.

### 3. Reveal phase is coach-driven, not learner-driven
The reveal dumps the full approach and the alignment/divergence analysis in one block. This is contrary to the Socratic ethos of the rest of the session. The user should discover the connections between their thinking and the paper's approach, not be told.

**Recommendation:** Split the reveal into two parts: (1) Present the paper's approach. (2) Ask "What do you notice? How does this compare to what you were thinking?" and let the user draw connections before the coach fills in gaps.

### 4. No personalized announcement
The "Coaching: *{title}*. Let's go." announcement is the same regardless of the user's interests. A one-sentence connection to the user's stated focus (from interests.toml) would increase engagement and help catch wrong-ID cases faster ("this doesn't sound related to my work" is easier to assess with context).

### 5. Missing feedback mechanism
There is no way for the user to rate the session or provide feedback on coaching quality. This data would be valuable for:
- Adjusting coaching preferences over time (auto-tuning difficulty/hint_style)
- Identifying which papers make good coaching candidates
- Improving the suitability assessment heuristic

### 6. Ambiguity around related-work questions during Socratic loop
The skill says "Do NOT reveal the paper's solution, method names, specific results." But what about related work? If the user asks "Has X been tried before?" the answer may be in the paper's related work section. The skill provides no guidance on whether to answer such questions, redirect them, or defer to after the reveal.

**Recommendation:** Add guidance: "Questions about the general field and related work may be answered if the information is widely known. Do not reveal content from the paper's own related work section if it would telegraph the approach."

### 7. Session JSON lacks metadata
The session file includes only the basics (arxiv_id, title, date, source, outcome, conversation). Missing fields that would be useful:
- `duration` (approximate session length)
- `num_exchanges` (how many Socratic loop turns)
- `hints_given` (count of explicit hints)
- `user_background` (snapshot from interests.toml at time of session)
- `coaching_preferences` (difficulty + hint_style used)

### 8. No handling of user wanting to coach the same paper twice
The skill writes to `.paperdojo/history/YYYY-MM-DD-<arxiv_id>.json`. If a user coaches the same paper on the same day, the file would be overwritten. On different days, separate files are created but there's no reference to the previous session.

**Recommendation:** Use a timestamp-based filename (e.g., `YYYY-MM-DD-HHMMSS-<arxiv_id>.json`) or append a sequence number. When starting a session, check for existing history and offer: "You've coached this paper before (2026-03-15). Start fresh or review your previous session?"

## Suggestions

Ordered by impact (highest first):

1. **Add pause/resume capability** -- Save session state with `outcome: "paused"` on user request. Resume when `/coach <same_id>` is invoked and a paused session exists. This is essential for complex papers that require offline thinking. *High impact, moderate effort.*

2. **Define "solved" criteria explicitly** -- When user says "I think I've got it," prompt them to state their approach. Assess against the paper's key insight(s) and mark outcome accordingly. This makes the solved/revealed distinction meaningful for tracking progress. *High impact, low effort.*

3. **Split reveal into present-then-reflect** -- First show the paper's approach, then ask the user to identify connections before the coach provides alignment/divergence analysis. Maintains Socratic spirit through the end. *High impact, low effort.*

4. **Add post-session feedback prompt** -- After the warm closing, ask "How was this session? Anything you'd adjust?" and save the response in the session JSON. Use this data to tune coaching preferences over time. *Medium impact, low effort.*

5. **Personalize the announcement** -- After "Coaching: *{title}*. Let's go." add a one-sentence connection to the user's interests.toml focus. Helps catch wrong IDs and increases engagement. *Medium impact, trivial effort.*

6. **Add related-work question guidance** -- Clarify in the skill spec how to handle questions about the general field vs the paper's own related work section during the Socratic loop. *Medium impact, trivial effort.*

7. **Enrich session JSON metadata** -- Add duration, num_exchanges, hints_given, user_background snapshot, and coaching_preferences to the session file. Enables /report to generate richer analytics. *Medium impact, low effort.*

8. **Handle repeat coaching of same paper** -- Use timestamp-based filenames and check for existing history on entry. Offer review of previous session or fresh start. *Low impact, moderate effort.*

9. **Add max-turns guidance for Socratic loop** -- The skill has no guidance on when to proactively suggest the reveal if the user hasn't triggered it. After N exchanges (configurable, default perhaps 8-10), the coach could gently suggest: "Want to keep going, or shall we see how the paper approaches this?" *Low impact, low effort.*

10. **Test the MCP fallback chain** -- The JIT install commands in Step 1 reference `arxiv-latex-mcp` and `arxiv-mcp-server` as pip packages. These package names should be verified. The fallback also assumes `python3` and `pip` are available, which may not be true on all systems. *Low impact, low effort to verify.*
